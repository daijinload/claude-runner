# 共通ルール（全レビュータスク必読）

このファイルは `claude-runner.sh plan` で自動生成されるレビュータスクが参照する観点定義のサンプル。

**書き方の規約**:
- `# H1` セクション = 全タスク共通の前文（このセクション）
- `## H2` セクション = 1 セクション = 1 並列レビュータスク（H2 を増やせば次の plan からタスクが増える）
- `###` 以降は共通ルール内のサブ見出しとして使う（H2 = タスクの法則を破らない）

このファイルは差替・追加・削除自由。プロジェクトのレビュー観点を H2 として書き足していく前提のテンプレート。

**H2 セクションの長さは自由**。下記の `code-quality` のように観点をしっかり並べてもよいし、`dead-code` のように数行で済む軽量チェックでも構わない。1 観点 1 セクションに分けて並列で走らせるのがこのスクリプトのコンセプト。

### 厳守事項

- **一切の推測を許さない。** 詳細に調査し、全ての指摘は明確な裏付け（コード逐語引用 + ルール逐語引用）まで確認する。
- **修正・コミット・push は禁止**（read-only）。発見だけして出力ファイルに記すのが任務。
- **質問は会話に出さず、出力ファイルの `## Questions` セクションに全て集約**する。後でまとめて回答する運用。
- **grep/rg 系の確認は AI 目視に頼らず、必ずシェルで実行**してシステム的に網羅確認する。
- 曖昧な指摘は severity を下げるか、出さない（critical は確信できる場合に限定）。

### レビュー対象差分

- `git diff origin/main...HEAD`（origin/main 分岐後の全コミット）
- + uncommitted 変更（`git diff` / `git status` で検出される未コミット分も含める）

差分が大きい場合は `git diff --stat` でファイル一覧 → 該当ファイル単位で `git diff` を見る。

### 出力フォーマット

各タスクの出力ファイル構成:

1. `## Summary` — verdict (🟢 PASS / 🔴 FAIL) + critical/warning/info 件数 + 対象言語・ファイル数
2. `## Findings` — 重要度順 (critical → warning → info)。各 finding は以下のフィールド:
   - severity (critical / warning / info)
   - category
   - `file:line`
   - title (短い要約)
   - detail (詳細説明)
   - 根拠ルール（参照したルール / ガイドラインの位置）
   - 根拠引用（短く中核部分）
   - コード引用（問題箇所の逐語引用）
3. `## Questions` — 途中で確認したいことが出たら箇条書きで全て書く。無ければ「なし」と書く

verdict 判定:
- critical が 1 件以上 → 🔴 **FAIL**
- 0 件 → 🟢 **PASS**

severity 判定基準（参考）:
- **critical**: 確実なバグ（null 参照 / 範囲外アクセス / 型不一致）/ セキュリティ脆弱性 / データ損失リスク / リソースリーク / 新規実装ファイルに対応するテストファイル完全欠落
- **warning**: コーディングルール違反 / 層の責務逸脱 / エラーハンドリング不備 / N+1 クエリ可能性 / テスト網羅性不足
- **info**: 軽微な改善 / Tips

## code-quality

差分の追加行 (`+`) に対して、標準的なコード品質の観点でレビューする。

観点:
- **命名**: 関数 / 変数 / 型が役割を表しているか。略語の濫用、副作用を隠す名前の禁止
- **責務**: 1 関数 / 1 ファイルの責務が肥大していないか、複数の関心事が混ざっていないか
- **エラーハンドリング**: error を握りつぶしていないか、catch して握りつぶし型の silent failure はないか

## dead-code
dead code がないか確認する

## duplicate-code
重複した関数を作成していないことを確認する
